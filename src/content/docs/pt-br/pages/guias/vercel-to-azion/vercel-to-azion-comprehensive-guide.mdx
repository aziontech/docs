# Migre da Vercel para a Azion

Uma migração de plataforma geralmente começa muito antes de o primeiro projeto ser implantado novamente ou o primeiro domínio de produção ser movido. Começa quando uma equipe percebe que fluxos de build, entrega, compute, IA, storage, segurança e observabilidade precisam de mais controle operacional do que o modelo atual oferece.

Para equipes que usam Vercel, esse momento costuma aparecer depois que as aplicações crescem além do fluxo original de deploy: projetos com múltiplos ambientes, URLs de preview, functions server-side, otimização de imagens, redirects, rewrites, dados no Edge Config, armazenamento Blob, integrações de IA, regras de firewall, controles de bots, dashboards de analytics e domínios de produção. Cada recurso pode funcionar individualmente, mas operar tudo como uma única plataforma de produção exige governança cuidadosa.

Para equipes que usam Vercel Projects, CI/CD and Preview Deployments, Content Delivery Network, Vercel Functions, Fluid Compute, AI Cloud, AI SDK, AI Gateway, Vercel Blob, Edge Config, Vercel Firewall, Web Application Firewall, Bot Management, BotID, Advanced Deployment Protection, Domains and DNS, Observability, Speed Insights, Web Analytics ou integrações do Marketplace, a Azion oferece recursos equivalentes por meio de Applications, Azion CLI, Preview Deployment, Functions, AI Inference, Azion AI Client, Object Storage, KV Store, Firewall, Web Application Firewall, Bot Manager, Edge DNS, Certificate Manager, Real-Time Metrics, Real-Time Events, Data Stream, Edge Pulse e Marketplace.

O motivo mais forte para migrar não é simplesmente substituir um fornecedor por outro. É consolidar deployment, entrega, compute, IA, storage, segurança e observabilidade em uma plataforma unificada projetada para aplicações distribuídas globalmente.

## Como funciona a migração da Vercel para a Azion

Migrações de plataforma tradicionais frequentemente exigem reconstruir a entrega da aplicação, retrabalhar functions, recriar configuração de ambiente e mover domínios de produção sob pressão. Essa abordagem aumenta o risco operacional, estende prazos e dificulta rollback.

A abordagem de migração da Azion preserva o comportamento da sua aplicação enquanto faz a transição para uma plataforma unificada:

1. **Caminho de migração incremental.** Comece com um projeto Vercel representativo, valide build, entrega, functions, storage, segurança e observabilidade na Azion e só então expanda para projetos adicionais.

2. **Lógica de aplicação preservada.** Rotas, redirects, rewrites, headers, variáveis de ambiente, functions server-side, arquivos de objeto, dados de configuração e políticas de segurança mapeiam para Azion Applications, Rules Engine, Functions, Object Storage, KV Store, Firewall e produtos de observabilidade com mudanças focadas.

3. **Modelo de plataforma unificado.** Em vez de gerenciar build, runtime, storage, segurança, analytics e observabilidade em superfícies separadas, a Azion reúne esses recursos com APIs, fluxos de Console e padrões de deployment consistentes.

## Estratégia de migração

A migração é organizada em torno das quatro categorias de produtos da Azion, permitindo que as equipes planejem e executem cada camada independentemente:

- **Build**: migre projetos, configurações de build, deployments de produção e preview, comportamento de CDN, redirects, rewrites, headers, otimização de imagens, functions, padrões de Fluid Compute e lógica de aplicações de IA.
- **Secure**: migre domínios, DNS, certificados TLS, regras de firewall, políticas WAF, controles de bots, postura de DDoS, regras de acesso a deployments e dependências de SSO.
- **Store**: migre arquivos Blob e dados do Edge Config para Object Storage e KV Store.
- **Observe**: migre dashboards de observabilidade, monitoramento de usuários reais, analytics, logs de functions, investigação de eventos e fluxos de exportação de dados.

:::note
Se você não precisa de tempo de inatividade próximo de zero, pode migrar em fases com janelas de manutenção. Isso permite parar escritas durante cada etapa de migração sem exigir sincronização paralela de dados.
:::

## Build

A categoria Build cobre configuração de projeto, deployment, configuração de entrega, roteamento, comportamento de runtime, Functions, otimização de imagens e workloads de IA. Comece aqui para recriar o comportamento da aplicação que os usuários veem antes de mover camadas de segurança, dados e observabilidade.

### 1. Configuração do projeto na Azion

O primeiro passo traz seu projeto Vercel para a Azion de uma forma familiar para equipes que implantam aplicações web modernas a partir do Git. Se você usou projetos Vercel, production deployments, preview deployments e presets de framework, já entende o padrão: conectar um repositório, definir configurações de build, implantar a saída e validar uma URL gerada.

A Azion segue um fluxo similar, mas consolida entrega de aplicações, Functions, regras, segurança, storage e observabilidade no mesmo contexto de plataforma.

#### Diferenças principais

| Aspecto | Vercel | Azion |
| :------ | :----- | :---- |
| **Recurso principal** | Project | Application + Workload |
| **Arquivo de config** | `vercel.json`, configuração de framework, configurações de projeto | `azion.config.js` e configuração via Console |
| **Fluxo de preview** | Preview deployments | Azion Preview Deployment e fluxos de CLI |
| **Roteamento e comportamento** | Rotas do framework, `vercel.json`, config de CDN | Applications, Rules Engine, Functions |
| **Observabilidade** | Observability, Speed Insights, Web Analytics | Real-Time Metrics, Real-Time Events, Data Stream, Edge Pulse |

#### Conecte seu repositório

1. Abra o [Azion Console](https://console.azion.com/).
2. Clique em **+ Create** > **Import from GitHub**.
3. Autorize o GitHub App da Azion.
4. Selecione o repositório que deseja migrar.

:::note
Mantenha o primeiro deployment intencionalmente pequeno. Não tente migrar todos os projetos Vercel, functions, domínios, Blob stores, valores de Edge Config, políticas de firewall, dashboards de analytics e integrações de IA de uma vez. Comece provando que uma aplicação representativa consegue construir e rodar na Azion.
:::

#### Configure seu build

A Azion detecta automaticamente seu framework e configura as definições de build. Sobrescreva o preset detectado em `azion.config.js`:

<Code lang="javascript" code={`
import { defineConfig } from 'azion'

export default defineConfig({
  name: 'my-vercel-migration',
  preset: 'nextjs', // Sobrescreva a auto-detecção se necessário
})
`} />

#### Faça deploy e verifique

Faça deploy pelo Azion Console ou CLI. Sua URL temporária da Azion segue este padrão:

<Code lang="text" code={`
https://xxxxxxxxxx.map.azionedge.net/
`} />

Valide o deployment antes de mover domínios de produção:

<Code lang="bash" code={`
curl -I https://xxxxxxxxxx.map.azionedge.net/
curl https://xxxxxxxxxx.map.azionedge.net/api/health
`} />

### 2. Convertendo configuração de build e deployment

Uma migração pode parecer bem-sucedida quando o build passa, mas falhar depois quando comportamento de runtime, variáveis de ambiente, redirects, cache ou otimização de imagens diferem da Vercel. Revise cuidadosamente a configuração do projeto em vez de tratar a migração como uma simples substituição de comandos.

#### Referência rápida de CLI

| Tarefa | Vercel CLI | Azion CLI |
| :----- | :--------- | :-------- |
| **Instalar** | `npm i -g vercel` | `curl -fsSL https://cli.azion.app/install.sh \| bash` |
| **Login** | `vercel login` | `azion login` |
| **Dev local** | `vercel dev` | `azion dev` |
| **Deploy preview** | `vercel` | `azion deploy` e fluxos de Preview Deployment |
| **Deploy produção** | `vercel --prod` | `azion deploy` |
| **Ver logs** | `vercel logs` | `azion logs` |
| **Rollback** | Rollback de deployment Vercel | `azion rollback` |

#### Inventário de configuração

Antes de recriar o projeto na Azion, faça inventário de:

* Projetos Vercel, times, production deployments e preview deployments
* Build commands, diretórios de saída, presets de framework e comandos de install
* `vercel.json`, configuração de framework, middleware e definições de rota
* Variáveis de ambiente entre produção, preview e desenvolvimento
* API routes, server actions, serverless functions e workloads de Fluid Compute
* Redirects, rewrites, headers, comportamento de cache e configurações de otimização de imagens
* Domínios, registros DNS, nameservers e status de certificados
* Blob stores, Edge Config stores, feature flags e experimentos
* Regras de firewall, controles WAF, proteções contra bots, controles de acesso a deployments e dependências de SSO
* Dashboards de observabilidade, analytics, alertas, fluxos de logs e integrações do Marketplace

### 3. Migrando variáveis de ambiente

Variáveis de ambiente contêm API keys, credenciais de banco de dados, segredos de autenticação, endpoints de terceiros, tokens de provedores de IA, feature flags e configurações específicas por ambiente. Migrá-las incorretamente causa falhas de runtime mesmo quando o deployment é bem-sucedido.

#### Diferenças principais

| Aspecto | Vercel | Azion |
| :------ | :----- | :---- |
| **Acesso** | `process.env.VARIABLE` | `Azion.env.get('VARIABLE')` |
| **Escopos** | Production, Preview, Development | Function instance e contexto de ambiente |
| **Secrets** | Variáveis de ambiente do projeto | Variáveis em Functions Instances |
| **Dados de configuração** | Variáveis de ambiente e Edge Config | Variables e KV Store |

#### Avalie suas variáveis

Antes de alterar código, identifique todas as variáveis em:

* Configurações de ambiente do projeto Vercel
* Variáveis de ambiente gerenciadas pela Vercel CLI
* Arquivos `.env` do framework usados em desenvolvimento local
* Chaves de provedores de IA e configurações de model gateway
* Tokens Blob e credenciais de storage
* IDs, tokens e chaves de Edge Config
* Configurações de ambiente CI/CD
* Configuração de runtime no código fonte

#### Crie variáveis na Azion

### 4. Migrando roteamento de CDN, redirects, rewrites e headers

O comportamento de roteamento da Vercel pode vir de rotas do framework, middleware, `vercel.json`, configurações do projeto e comportamento de CDN. Azion Applications e Rules Engine fornecem controles equivalentes para roteamento, redirects, rewrites, headers, seleção de origem, comportamento de cache e manipulação de request ou response.

#### Diferenças principais

| Aspecto | Vercel | Azion |
| :------ | :----- | :---- |
| **Configuração de roteamento** | Rotas do framework e `vercel.json` | Applications + Rules Engine |
| **Redirects** | `redirects` no `vercel.json` ou na configuração do framework | Comportamento de redirect no Rules Engine |
| **Rewrites** | `rewrites` no `vercel.json` ou na configuração do framework | Rules Engine e Functions |
| **Headers** | `headers` no `vercel.json` ou na configuração do framework | Comportamentos de request/response no Rules Engine |
| **Seleção de origem** | Entrega gerenciada pela Vercel | Connectors + Rules Engine |

#### Exemplo de migração de redirect

<Code lang="json" code={`
// Antes: vercel.json
{
  "redirects": [
    {

### 1. Migrando Vercel Blob para Object Storage

Vercel Blob armazena arquivos de aplicação como imagens, documentos, vídeos e uploads de usuários. Azion Object Storage é compatível com S3 e pode ser usado como origem para Applications.

#### Diferenças principais

| Aspecto | Vercel Blob | Azion Object Storage |
| :------ | :---------- | :------------------- |
| **Modelo de dados** | Objetos Blob | Buckets e objetos |

| **Arquivos privados** | Token e padrões de URL | Permissões de bucket/objeto e lógica de aplicação |

#### Atualize a configuração

<Code lang="javascript" code={`

  access: 'public'

import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';

  region: 'us-east-005',

Use ferramentas como `s3cmd`, `rclone` ou AWS CLI para mover objetos:

| Tarefa | Padrão de comando |
| :----- | :---------------- |
| Listar buckets | `s3cmd ls` |
| Fazer upload de objeto | `s3cmd put file.png s3://my-bucket/` |
| Fazer download de objeto | `s3cmd get s3://my-bucket/file.png` |

## Troubleshooting

### Problemas comuns

| Problema | Causa provável | Solução |
| :------- | :------------- | :------ |
| **Build falha na Azion** | Preset de framework, comando de install ou diretório de saída difere da Vercel | Verifique `azion.config.js`, scripts do pacote e saída do build |
| **Variáveis de ambiente não encontradas** | Variáveis não foram configuradas na Function instance | Confirme que as variáveis existem e que o código usa `Azion.env.get()` |
| **Redirects ou rewrites se comportam de forma diferente** | Padrões de rota da Vercel foram convertidos para regex incorreto | Teste capture groups e verifique critérios do Rules Engine |
| **API routes falham** | A assinatura da Function ou o parsing de request ainda presume runtime da Vercel | Reescreva handlers usando `fetch(request)` e Web APIs padrão |
| **Imagens não são otimizadas** | URLs de imagem da Vercel não foram convertidas ou Image Processor está desabilitado | Habilite Image Processor e atualize padrões de URL de imagem |
| **Comportamento de cache mudou** | Comportamento de cache do framework, TTL ou cache key não foi mapeado | Revise Cache Settings, Advanced Cache Key e Rules Engine |
| **Arquivos Blob ausentes** | Exportação ou importação de objetos incompleta | Exporte arquivos novamente, verifique object keys e valide permissões do bucket |
| **Dados do Edge Config ausentes** | Namespace KV ou script de migração incompleto | Recrie chaves, valide nomes de namespace e teste valores padrão |
| **Regras de firewall bloqueiam usuários válidos** | Lógica de regra ou listas de IP diferem da Vercel | Execute em modo monitorado, compare eventos e ajuste critérios |
| **Lacuna de analytics após cutover** | Dependências de analytics da Vercel não foram substituídas | Configure Edge Pulse, Real-Time Metrics, Real-Time Events e Data Stream |
| **Certificado não está ativo** | Validação de domínio ou associação de certificado incompleta | Confirme o status do certificado e a associação ao workload antes do cutover de DNS |

## Principais vantagens após a migração

Migrar da Vercel para a Azion funciona melhor quando a equipe trata o primeiro cutover como um padrão de migração repetível. O objetivo imediato é continuidade: usuários continuam acessando os mesmos domínios, rotas críticas continuam funcionando, controles de segurança permanecem ativos e observabilidade está pronta antes da movimentação de tráfego.

O valor maior vem depois da migração, quando as equipes operam build, entrega, compute, IA, storage, segurança e observabilidade em um único modelo de plataforma.

| Área | Vantagem após a migração | O que isso significa na prática |
| :--- | :----------------------- | :------------------------------ |
| **Estratégia de migração** | Caminho de migração incremental | As equipes podem migrar um projeto Vercel por vez e validar cada camada independentemente. |
| **Modelo de plataforma** | Plataforma unificada de aplicações | Recursos de Build, Secure, Store e Observe são gerenciados por um modelo de plataforma conectado. |
| **Build** | Deployment moderno de aplicações | Applications podem ser implantadas a partir do GitHub ou da CLI com configuração da Azion. |
| **Build** | Controle de aplicação baseado em regras | Rules Engine gerencia redirects, rewrites, headers, comportamento de cache, roteamento e lógica de request/response. |
| **Build** | Execução distribuída de functions | Functions substituem padrões de Vercel Functions e Fluid Compute para APIs, personalização, orquestração de IA e integrações. |
| **Build** | Cache e otimização de imagens integrados | Cache, Tiered Cache, Real-Time Purge e Image Processor são configurados como parte da camada de entrega. |
| **IA** | Workloads de IA integrados à entrega | AI Inference e Functions oferecem suporte a chamadas de modelo, roteamento e lógica de aplicação junto com observabilidade. |
| **Secure** | Cutover controlado de domínio e TLS | Workloads, Edge DNS e Certificate Manager podem ser preparados antes de trocar tráfego de produção. |
| **Secure** | Segurança integrada de aplicações | WAF, Bot Manager, Network Shield, padrões de rate limiting e regras de Firewall protegem workloads em conjunto. |
| **Store** | Object Storage compatível com S3 | Workflows de Vercel Blob podem ser migrados usando ferramentas e SDKs compatíveis com S3. |
| **Store** | KV para dados de baixa latência da aplicação | Casos de uso de Edge Config podem ser migrados para KV Store para configuração, flags, experimentos, redirects e estado leve. |
| **Observe** | Métricas em tempo real para visibilidade de produção | Real-Time Metrics fornece dashboards e acesso GraphQL para dados de tráfego, cache, latência e erro. |

Com a Azion, as equipes podem construir e executar aplicações em infraestrutura globalmente distribuída enquanto combinam compute, entrega, IA, storage, segurança e observabilidade em um único ambiente. Isso reduz a fragmentação que costuma aparecer quando aplicações modernas dependem de muitos serviços e padrões de configuração separados.

### Comece com um projeto pequeno

A melhor forma de começar não é escolher o projeto Vercel mais complexo do seu portfólio. Comece com um projeto significativo o suficiente para validar o caminho de migração, mas pequeno o suficiente para mover com rapidez e segurança.

Escolha um projeto que inclua partes representativas da sua arquitetura: um domínio semelhante ao de produção, algumas rotas, redirects, headers, talvez uma function, uma dependência de Blob, uma dependência de Edge Config e um fluxo de analytics. Use esse projeto para validar o workflow, documentar o processo e identificar padrões internos que sua equipe pode reutilizar.

A partir daí, expanda gradualmente. Migre roteamentos mais complexos. Mova functions adicionais. Traga storage e dados de configuração. Adicione observabilidade. Revise regras de segurança. Depois, prepare cutovers de produção com mais confiança.
